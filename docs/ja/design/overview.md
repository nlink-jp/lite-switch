# 設計概要

## 目的

lite-switch は自由形式のテキストを、LLM を使って事前定義されたタグのいずれかに分類します。
Unix フィルターとして設計されており、stdin を読んで stdout に1つのタグを書き出します。

## 設定管理

以下の優先順位（高い順）で値を解決します:

1. CLI フラグ（`-config`, `-switches`）
2. 環境変数（`LITE_SWITCH_API_KEY`, `LITE_SWITCH_BASE_URL`, `LITE_SWITCH_MODEL`）
3. 設定ファイル（`~/.config/lite-switch/config.toml`）
4. コンパイル時デフォルト値

### 2ファイル構成

| ファイル | フォーマット | 内容 | 場所 |
|---------|------------|------|------|
| `config.toml` | TOML | API 認証情報、モデル、タイムアウト | `~/.config/lite-switch/` |
| `switches.yaml` | YAML | タグ・説明のペア | プロジェクトディレクトリ |

2ファイルに分けているのは、セキュリティプロファイルと変更頻度が異なるためです。
スイッチファイルには機密情報が含まれないため、バージョン管理に含めて問題ありません。
スイッチファイルに YAML を採用しているのは、リスト形式のデータとして TOML の
配列テーブル（`[[switches]]`）より読みやすいためです。

## 分類パイプライン

```
stdin
  └─ readInput()           — サイズ制限付き読み込み、空白トリム
       └─ WrapUserInput()  — ノンス付き XML タグでラップ（プロンプトインジェクション対策）
            └─ Classify()  — リクエスト構築、LLM 呼び出し、タグ抽出
                 └─ extractTag()  — 4段階フォールバック:
                      1. ツールコールの結果
                      2. コンテンツ中の JSON {"tag": "..."}
                      3. コンテンツ中のタグ文字列
                      4. 最後のスイッチ（キャッチオール default）
  └─ stdout: タグ1つ
```

## LLM との対話

分類器はファンクションコール / ツール使用 API（`tools` + `tool_choice: "required"`）を使います。
これによりモデルが構造化レスポンスを返すことを強制し、ほとんどの場合でフリーテキスト解析を不要にします。

ツールコールに対応していないモデルに対しては、フォールバックチェーン（JSON → テキスト → デフォルト）が
常に使用可能な結果を返すことを保証します。

## エンドポイント URL の扱い

`base_url` は `http://host` と `http://host/v1` の両形式を受け付けます。
クライアントが `/chat/completions` または `/v1/chat/completions` を適切に付加します。

## リトライポリシー

一時的なエラー（ネットワーク障害、HTTP 429、HTTP 5xx）は指数バックオフでリトライします。
リトライ不可能なエラー（429 以外の 4xx）はすぐに失敗します。
